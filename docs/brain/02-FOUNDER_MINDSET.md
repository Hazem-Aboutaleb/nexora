# FOUNDER MINDSET

## Purpose

This document defines how decisions are made in Nexora.

Any AI assistant, developer, or future team member should understand this mindset before writing code.

---

# Primary Goal

The goal is not to learn Flutter.

The goal is to build a real software company.

Technology is only a tool.

---

# Learning Philosophy

The Founder prefers understanding over memorization.

For every new technology:

1. What is it?
2. Why do we need it?
3. Why did we choose it?
4. When should we use it?
5. What are the alternatives?

Implementation comes after understanding.

---

# Development Philosophy

Always build from the foundation upward.

Project Order:

Vision

↓

Planning

↓

Documentation

↓

Architecture

↓

Development

↓

Testing

↓

Deployment

Never skip a stage.

---

# MVP Philosophy

Always build the smallest usable product.

Avoid Feature Creep.

A finished small product is more valuable than an unfinished large product.

---

# Decision Making

Before adding any feature ask:

Does it solve a real problem?

Does it make the product smarter?

Can it wait until after MVP?

If the answer is "Yes, it can wait"

Do not build it now.

---

# Communication Rules

One task at a time.

No unnecessary explanations.

No repeated information.

Explain only when needed.

Stay focused on the current Sprint.

---

# Documentation Rules

Documentation is mandatory.

Important discussions should become documentation.

Knowledge should never remain inside chat history.

GitHub stores the project.

Documentation stores the knowledge.

---

# Engineering Philosophy

Prefer clean architecture over shortcuts.

Prefer maintainability over speed.

Prefer clarity over clever code.

Every folder should have a purpose.

Every file should have a responsibility.

---

# AI Collaboration Rules

The AI acts as:

- Technical Co-Founder
- Software Architect
- Senior Engineer
- Code Reviewer

The Founder owns:

- Product Vision
- Business Direction
- Final Decisions

---

# Product Rules

Never replace customer hardware.

Never overcomplicate the product.

Never build unnecessary features.

Business value comes before technology.

Customer experience comes before implementation details.

---

# Git Rules

Commit frequently.

Write meaningful commit messages.

Push stable work only.

GitHub is always the official project history.

---

# Long-Term Goal

Build Nexora into a scalable software company.

The project should remain understandable after years of development.

New developers should be able to understand the project by reading the documentation before reading the code.
